Writes are O(S). Reads use a sawtooth strategy.

Operation Cost

Writes are strictly constant time (O(S)) due to atomic appends. Reads involve traversing the delta chain, making them linearly more expensive until a snapshot resets the cost.

To prevent infinite read degradation, LedgerDB enforces a full document snapshot every 32 transactions. This creates a "sawtooth" performance profile that guarantees an upper bound on read latency.

Frequently Asked Questions

Why build a database on Git?
Git provides out-of-the-box cryptographic verification (Merkle trees), distributed replication, and audit trails. By using Git as the storage engine, LedgerDB inherits these properties natively without complex consensus algorithms.
How does concurrency work without a central server?
LedgerDB uses optimistic concurrency control on a per-document basis. Writers check the `HEAD` pointer before writing. If `HEAD` has moved since the read, the write fails and must be retried.
Is it SQL compatible?
Yes. LedgerSQL acts as a derived layer that projects the JSON documents into a local SQLite instance. This allows you to run complex SQL queries and joins on your data while keeping the Git ledger as the single source of truth.
